# ServEcosys Root Project

## 定位

ServEcosys 是一个基于 Linux 构建、面向 PC、移动端及多种设备形态的开源根项目（Root Project）。其定位类似于 AOSP（Android Open Source Project），旨在提供一个自由、开源、集百家之长的通用操作系统底座，打破现有商业系统的垄断，建立"底层开源共建、中层公共服务、上层应用繁荣"的数字生态。

**核心纲领：** 继承 Linux 的自由与开放精神，以用户体验需求为中心，为用户服务。系统的一切设计——从架构到权限、从生态到恢复——都以用户的掌控权、选择权和安全感为出发点。

## 核心架构：三层纵深，多设备适配

### 底基系统（Base System）— UEFI 层

底基系统是 ServEcosys 的最底层基石，一个最小化、可独立运行的开发与恢复环境，作为 UEFI 应用（payload）存在于 UEFI 固件或 ESP 分区中。它提供：

- **直接硬件访问能力**：绕过内核抽象，直接操作 CPU 寄存器、内存映射 I/O、中断控制器等，适合开发底层驱动、Bootloader、固件或进行硬件调试。

- **最小化内核无关运行时**：不依赖完整 Linux 内核，仅包含必要的自举代码、内存管理原语和串口/网络控制台。可作为开发者的"救生舱"——当主操作系统崩溃或需要调试内核时，从此环境启动。

- **操作系统加载与管理**：能够加载、校验并启动主 Linux 内核及 initramfs，也可用于刷写系统镜像、管理 Btrfs 子卷、修复启动链。本质上是一个高级的、可编程的引导加载程序 + 恢复系统。

- **完全开放的可编程性**：底基系统本身是开源的，开发者可以修改其代码，甚至用它来开发一个全新的操作系统内核。它是"最原初的开发环境"这一概念的物理实现。

用户通过底基系统拥有对设备的完整控制权：可以删掉整个 ServEcosys、刷自己的内核、换自己的 OS、修改分区表、重刷固件——不需要任何人的许可。这是"为用户服务"在硬件层面的终极兑现。

### 唯一 Linux 内核

- 基于 Linux LTS 主线深度定制，保持上游长期兼容性。

- 内核中央仅保留进程调度、内存管理、网络协议栈和核心安全钩子（LSM）等最小化基础能力。

- 设备驱动剥离为可插拔的独立模块，按功能归属到 PC 设备适应模块集、移动设备适应模块集，并可扩展至 IoT、嵌入式等其他设备模块集。

- 原生集成 SELinux 作为默认安全模块，编译期开启严格的策略加载和审计机制，不可在运行时降级或旁路。

- 内置硬件指纹探测模块：首次启动时遍历设备树和总线信息，生成硬件指纹快照并签名存储；后续启动直接读取该快照，仅加载所需模块，跳过全量探测以缩短启动时间。

### 后端安全域（SED）

后端安全域是系统安全策略的单一执行点，运行于不受限的 SELinux 上下文（sys_dom_t）中。它负责：

- **SELinux 策略执行**：所有安全决策集中在此域完成，前端域或第三方进程无权修改策略或审计规则。

- **硬件抽象与驱动管理**：仅 SED 中的系统服务拥有对块设备、总线、传感器等物理资源的直接访问权。

- **权限判定与审计**：任何从前端域发起的特权操作均以 IPC 请求形式提交至 SED 进行实时裁决。

- **OIPES 客户端代理**：管理与中立公共服务端的加密连接、本地令牌存储、推送消息过滤和 AI 请求的本地预处理。

SED 中的系统服务均编译为标准的 ELF 可执行文件，使用 .smle（System Module Loadable Executable）后缀作为逻辑标识，并以专用目录（/system/backend/bin）和 SELinux 文件标签双重约束。其进程禁止与前端域共享内存页，且 IPC 通道固定为单向响应模式——前端可发起请求，但无法主动访问 SED 内部的任何数据结构。

### 前端交互域（UID）

前端交互域负责用户界面渲染与交互处理，运行于受限的 SELinux 上下文（uid_dom_t）中。它包含：

- **显示服务与输入管理**：兼容 PC 端键盘/鼠标、移动端触控/手写笔，以及其他设备形态的输入方式，统一抽象为逻辑输入事件。

- **应用沙箱运行时**：所有原生第三方应用以 .ssle（Subsystem Module Loadable Executable）后缀的 ELF 文件形式部署于 /system/frontend/apps/，每个应用启动时自动分配独立的 SELinux 标签和文件系统命名空间，默认无法访问网络、持久存储或外设，权限由用户在安装时授予并经 SED 运行时裁决后动态下发。

- **系统界面层**：如状态栏、通知中心、多窗口管理器，以系统签名的 .smle 服务形式运行，但其交互渲染面仍置于 UID 中以确保响应延迟可控。

## 启动顺序

1. **固件阶段**：UEFI 或设备固件加载签名的引导程序（或直接进入底基系统）。
2. **底基系统阶段（可选）**：引导程序可配置为直接启动底基系统，或跳过它直接进入主内核。若进入底基系统，用户获得最小化控制台，可执行硬件诊断、系统修复、内核加载等操作。底基系统亦可选择性地加载并校验主内核。
3. **主内核初始化**：内核中央完成内存、调度、设备树解析等基础初始化，将控制权移交 initramfs。
4. **硬件适配**：initramfs 中的初始化脚本根据缓存决定加载的模块集，执行必要的设备驱动绑定。
5. **根文件系统切换**：挂载 Btrfs 根子卷，执行 pivot_root 进入真实系统环境。
6. **后端安全域启动**：systemd（或等效初始化系统）以强隔离 SELinux 上下文启动 SED 核心服务，包括 SELinux 策略管理器、硬件抽象层守护进程和 OIPES 客户端代理。此时系统已具备安全决策能力，但用户界面尚未呈现。
7. **前端交互域启动**：SED 确认自身初始化完成后，通过安全 IPC 触发 UID 初始化——创建沙盒命名空间，启动显示服务并呈现登录界面。第三方应用的自动启动需由 SED 根据用户预设的权限策略逐一放行。

## 权限体系：0–11 级权限阶梯

ServEcosys 的全栈权限体系建立在同一棵 SELinux 策略树上，LSM hooks 为根。权限分为 0 到 11 级，每一级代表对系统底层不同程度的控制能力：

| 级别 | 权限说明 |
|------|----------|
| 0 | 只读、只写、只读——执行权限 |
| 1 | 应用沙盒权限 |
| 2 | 普通用户 / 系统应用权限 |
| 3 | 进阶调试权限 |
| 4 | BL 锁解锁权限、特权文件权限 |
| 5 | Root 分能力权限、自定义恢复环境权限 |
| 6 | 模块加载 Root 权限 |
| 7 | 系统内核加载 Root 权限（更深） |
| 8 | SELinux 控制权限 |
| 9 | 内核模块加载权限 |
| 10 | 自定义内核权限 |
| 11 | 引导加载程序 / 启动链控制权限 |

**关键设计原则：**

- **4 级 BL 解锁是一道自由之门**：解锁后，用户可以自签 SED 策略、自编内核，在自己的硬件上运行任何自己信任的代码。这是"为用户服务"的终极体现——你的设备，你做主。

- **8 级和 11 级的具体控制方式由项目方定义，但用户通过底基系统拥有对设备的完整物理控制权**：用户不需要官方签名密钥——底基系统给了用户掀桌子的路。用户可以选择留在官方生态享受无缝服务，也可以选择完全自治，两者都是自由。

- **用户自签链上无法连接 OIPES 高等级服务**，但这并非惩罚，而是技术必然——因为自签链的信任根不在官方信任锚点之内。

这套权限阶梯的本质是：**把选择权交给用户，同时把安全底线锁在透明、可审计的代码和机制中。** 用户永远不会被强制降级，也永远不会被暗中剥夺对自己设备的控制权。

## IPC：最小化权能总线，可换传输层

双域之间的通信通过一个定制的最小化 IPC 总线实现。该总线基于权能（capability）模型设计：

- 前端域中的每个应用或服务在启动时获得一个最小化权能令牌。
- SED 仅处理携带有效令牌的请求，且每次请求均附带调用链的审计标记。
- 总线实现完全在 Linux 用户空间通过 binderfs 改造和 SELinux 钩子实现，避免引入第二内核。

IPC 是模块化且可换的：包含一个极简分发器（单实现，不可换，负责验令牌和转发），一个可插拔模块层（按设备剪裁），以及一个可换传输层（生产版用 binderfs，调试版用 unix socket，极限性能版用 io_uring）。

## .ssle：原子化应用单元

.ssle 不是单个 ELF，而是一个启动 ELF 加上多个业务 ELF 压缩合并而成的单体包。一个 .ssle 包就是一个权限单元，SED 进行包级裁决而非 per-ELF 裁决。包内多 ELF 共享同一 SELinux 标签，同包内 ELF 之间直接交流，跨包才走 IPC。

这是"Linux 原生小改"的真正落地：ELF 格式不动（ET_DYN/ET_EXEC），libc 不变，开发者只需改打包方式和 IPC 调用路径。每一个 ELF 都是一个功能，包括 .smle——系统服务和用户应用在 ELF 格式层面没有区别，只靠部署路径、SELinux 标签和签名密钥区分。

## OIPES：开放生态服务，拒绝绑定

OIPES（操作系统互联网公共生态服务）提供与生态系统的标准化连接。它采用严格的服务端-客户端解耦模型：

- **服务端**：由中立非营利组织独立运营，提供身份认证服务、基础 AI 模型推理接口、系统级推送网关以及可选的云存储中继。服务端组件不在本项目的代码库中，仅提供开放 API 规范。

- **客户端**：以 .smle 服务形式运行于后端安全域中，负责管理与服务端的加密连接、本地令牌存储、推送消息的优先级过滤和 AI 请求的本地缓存与预处理。客户端代码完全开源，用户可随时查看或关闭 OIPES 通信，或者替换为兼容的第三方服务端点。

**操作系统不捆绑任何特定的云服务供应商。** 生态服务的选择权完全由用户和部署方掌握。这是"为用户服务"在生态层面的兑现。

## 系统自愈：Btrfs 原子快照，三层恢复

ServEcosys 使用 Btrfs 文件系统配合多子卷布局：

- `@system`：系统只读基准子卷，包含所有系统级二进制文件。
- `@data`：用户数据子卷，与系统卷完全独立。
- `@snapshots`：自动快照存储子卷，每次系统更新或安装应用前自动创建原子快照。

恢复机制分三层：

- **轻症**（更新滚挂、驱动炸、策略写崩）：在引导程序中直接选择上一个健康快照启动，10 秒内恢复系统，不触及 @data 用户数据。

- **中症**（文件系统元数据严重损坏或快照链失效）：从独立隐藏分区 /dev/recovery 引导，执行完整系统重装，同时保留用户数据子卷。

- **重症**（recovery 分区也损坏，或需要底层硬件操作）：从底基系统（UEFI 层）启动，直接访问存储设备，手动修复分区表、重刷固件或重新部署操作系统。底基系统可从 USB/SD 卡等外部介质加载，确保终极恢复能力。

所有快照和恢复操作均为原子性，避免因写入中断导致的系统半更新状态。

## 项目目录结构

```
/servecosys/                  # 项目根目录 (Root Project)
├── base/                     # 底基系统源码与构建脚本
│   ├── core/                 # 底基系统核心（最小化运行时）
│   ├── drivers/              # 底基系统可直接调用的硬件驱动（可选）
│   └── tools/                # 恢复、刷写、调试工具
├── boot/                     # 引导启动程序链
├── kernel/                   # 主内核文件夹
│   ├── core/                 # 内核中央
│   └── modules/              # 内核模块 (pc/, mobile/, probe/, ...)
├── backend/                  # 后端文件夹 (.smle)
│   ├── sub_kernel/           # 后端子内核
│   ├── security/             # 权限控制台与 SELinux
│   └── oipes/                # 公共生态服务
├── frontend/                 # 前端子内核文件夹 (.ssle)
│   ├── ui/                   # 界面层
│   └── apps/                 # 原生第三方应用
├── restore/                  # 恢复文件区 (factory_image)
└── scripts/                  # 构建与快照管理脚本
```

## 开发者体验：低门槛接入，高自由度发挥

- ✅ ELF 格式不动，libc 不变：现有 Linux 原生应用可低成本迁移。
- ✅ ssle-packer 工具链：自动处理打包、权限扫描和签名。
- ✅ 包级裁决减少 IPC 调用量：同包内 ELF 直连，无需频繁跨域通信。
- ✅ OIPES 开源免绑厂：不强制接入任何特定云服务。
- ✅ 底基系统开发环境：开发者可直接在底基系统上编写和测试内核模块、驱动程序，甚至从头构建一个新的操作系统内核，无需担心破坏主系统。底基系统提供串口/GDB 调试支持，是内核开发的理想平台。
- ✅ QEMU 多架构仿真 + 10 秒回滚：让开发者敢于试错，不怕搞坏系统。
- ✅ SELinux 审计日志：调试门槛虽比 logcat 略高，但提供了更强的安全保证。
