# 权限控制台与 SELinux

ServEcosys 安全策略执行点。

## 根本纲领

**以用户主权为设计原则，以用户体验需求为落地基准，为用户服务。**

权限体系的设计必须体现用户主权，让用户真正掌控自己的设备。

---

## 权限阶梯 (0-11 级)

| 级别 | 名称 | 权限说明 | 控制权 |
|------|------|----------|--------|
| 0 | READONLY | 只读/只写/只执行 | 系统 |
| 1 | SANDBOX | 应用沙盒 | 系统 |
| 2 | USER | 普通用户/系统应用 | 系统 |
| 3 | DEBUG | 进阶调试 | 系统 |
| 4 | BL_UNLOCK | BL 解锁/特权文件 | 用户 |
| 5 | ROOT_SPLIT | Root 分能力/自定义恢复 | 用户 |
| 6 | MODULE_ROOT | 模块加载 Root | 系统 |
| 7 | KERNEL_ROOT | 内核加载 Root | 系统 |
| **8** | **SELINUX** | **SELinux 控制** | **用户** ⭐ |
| 9 | KMOD_LOAD | 内核模块加载 | 系统 |
| 10 | CUSTOM_KERNEL | 自定义内核 | 用户 |
| **11** | **BOOTLOADER** | **引导加载程序/启动链** | **用户** ⭐ |

## 核心原则

### ⭐ 8 级和 11 级由用户完全控制

这是**以用户主权为设计原则，以用户体验需求为落地基准，为用户服务**的根本体现。

- **8 级（SELinux 控制）**：用户可以自签 SELinux 策略
- **11 级（引导加载程序）**：用户可以自签引导加载程序

**这意味着：**
- ✅ 用户不需要项目维护者的许可
- ✅ 用户不需要独立审计方的批准
- ✅ 用户可以在自己的硬件上运行任何自己信任的代码

### 4 级 BL 解锁：自由之门

解锁后：
- 用户可以自签 SED 策略
- 用户可以自编内核
- 用户可以在自己的硬件上运行任何自己信任的代码

这是"为用户服务"的终极体现——**你的设备，你做主**。

### 自签链的限制

用户自签链上无法连接 OIPES 高等级服务，但这：
- ❌ **不是惩罚**
- ✅ **是技术必然**

因为自签链的信任根不在官方信任锚点之内。

**用户的选择：**
1. 留在官方生态 → 享受无缝服务
2. 完全自治 → 运行自己的代码

**两者都是自由。**
